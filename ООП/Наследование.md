# Наследование

**Наследование** — это механизм объектно-ориентированного программирования, позволяющий создавать новые классы на основе существующих. Оно реализует отношение is-a («является»), то есть производный класс является разновидностью базового.

По умолчанию все классы неявно наследуются от базового класса **Object**, даже если это не указано явно.

## Ограничения наследования (на примере C#):
- Не поддерживается множественное наследование классов — класс может наследоваться только от одного базового класса.
- При создании производного класса необходимо учитывать модификатор доступа базового класса: производный класс должен иметь такой же или более строгий уровень доступа.
- Если базовый и производный классы находятся в разных сборках (проектах), наследование возможно только от класса с модификатором public.
- Класс, объявленный с модификатором sealed, не может иметь наследников.
- Нельзя наследоваться от статического класса.

## Особенности и ограничения подхода:
Наследование используется для построения иерархий связанных объектов и тесно связано с полиморфизмом. Однако оно имеет ряд недостатков:
- Создаёт сильную связанность между базовым и производными классами.
- Изменения в базовом классе могут повлиять на все производные классы и нарушить их работу.
- Производный класс может нарушить инварианты базового класса.
- Наследование плохо учитывает будущие изменения системы.

Важно понимать, что **наследование** само по себе не является основным инструментом повторного использования кода — для этого чаще применяются **композиция** и **агрегация**.

## Когда использовать наследование:

Наследование целесообразно применять в случаях, когда:
- между классами действительно существует отношение «является» (is-a)
- структура иерархии относительно стабильна и не предполагает частых изменений.

**Примеры:** Stream, Exception.

## Проблема наследования

Рассмотрим классический пример использования наследования:

```
abstract class Animal
{
    public abstract void Speak();
    public abstract void Walk();
    public abstract void Swim(); // Не все животные могут плавать
}

class Elephant : Animal
{
    public override void Speak()
    {
        Console.WriteLine("Ту-ту");
    }

    public override void Walk()
    {
        Console.WriteLine("Топ топ");
    }

    public override void Swim()
    {
        // ??? Слон не обязан уметь плавать
    }
}

class Dog : Animal
{
    public override void Speak()
    {
        Console.WriteLine("гаф");
    }

    public override void Walk()
    {
        Console.WriteLine("Топ топ");
    }

    public override void Swim()
    {
        Console.WriteLine("Брызг-брызг");
    }
}

class Penguin : Animal
{
    public override void Speak()
    {
        Console.WriteLine("ХЗ");
    }

    public override void Walk()
    {
        Console.WriteLine("Топ топ");
    }

    public override void Swim()
    {
        Console.WriteLine("Бульк...");
    }
}
```
### В чём проблема?

Базовый класс ```Animal``` задаёт поведение ```Swim()```, которое не является универсальным для всех животных. 

**В результате:**

- Некоторые классы (например, Elephant) вынуждены реализовывать метод, который им не нужен.
- Возникают «пустые» или некорректные реализации.
- Нарушается принцип корректного моделирования предметной области.

**Что это нарушает?**

Такой подход нарушает принцип подстановки Лисков (LSP) — объекты производного класса должны корректно заменять объекты базового класса без изменения ожидаемого поведения.

### Вывод

Не стоит помещать в базовый класс поведение, которое не является общим для всех наследников.

В подобных ситуациях лучше:

- выносить поведение в отдельные интерфейсы (например, ISwimmable)
- использовать композицию вместо наследования.
