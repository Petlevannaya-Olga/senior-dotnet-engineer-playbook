# Вопросы к собеседованию 

<details>
<summary><b>Чем инкапсуляция отличается от сокрытия данных?</b></summary>

> **Сокрытие данных** — это механизм ограничения доступа к внутреннему состоянию объекта (например, с помощью `private`, `protected`). Оно предотвращает прямое вмешательство извне.
>
> **Инкапсуляция** — более широкое понятие: объект объединяет данные и методы для работы с ними и самостоятельно контролирует корректность своего состояния.

### Ключевое различие

- сокрытие данных — это **инструмент (механизм)**;
- инкапсуляция — это **принцип проектирования**.

Сокрытие данных является частью инкапсуляции и используется для её реализации.

Инкапсуляция не просто ограничивает доступ, но и:

- задаёт правила работы с данными (валидация, проверки);
- гарантирует, что объект всегда находится в корректном состоянии;
- скрывает детали реализации, предоставляя только необходимый интерфейс.

Пример:

```csharp
class BankAccount
{
    private double balance; // сокрытие данных

    public void Deposit(double amount)
    {
        if (amount > 0) // контроль состояния (инкапсуляция)
        {
            balance += amount;
        }
    }

    public double GetBalance()
    {
        return balance;
    }
}
```
</details>

<details>
<summary><b>Что обычно нарушает инкапсуляцию в C#?</b></summary>

- публичные поля;
- свойства с `public set`, если объект должен защищать свои инварианты;
- возврат изменяемых коллекций напрямую;
- передача наружу ссылок на внутреннее изменяемое состояние;
- выдача ссылки на внутреннее состояние через `ref return`.

### Дополнительно

- отсутствие валидации в сеттерах или методах, изменяющих состояние;
- использование `internal` / `protected` без необходимости, если это расширяет доступ к деталям реализации;
- изменение состояния объекта извне через переданные или возвращённые зависимости;
- чрезмерное использование `public` вместо более строгих модификаторов доступа;
- отсутствие неизменяемости там, где она уместна: `readonly`, `init`, immutable-объекты;
- раскрытие деталей реализации через публичный интерфейс класса — ситуация, когда класс предоставляет наружу информацию о том, как он устроен внутри (используемые структуры данных, алгоритмы, технические шаги обработки), а не только то, какое поведение он предоставляет.
- использование публичных статических полей для изменяемого состояния.

Такие практики позволяют менять состояние объекта в обход его правил, из-за чего сложнее поддерживать инварианты и сопровождать код.

</details>

<details>
<summary> <b>Как безопасно вернуть коллекцию?</b></summary>

```csharp
private readonly List<Item> _items = new();

// Нежелательно: внешний код получает прямой доступ
// к внутренней изменяемой коллекции
public List<Item> Items => _items;

// Хорошо: коллекцию можно только читать
public IReadOnlyList<Item> Items => _items;

// Если нужен независимый снимок текущего состояния
public Item[] GetItems() => _items.ToArray();
```

### Важное уточнение

```IReadOnlyList<T>``` не делает элементы коллекции неизменяемыми. Этот интерфейс лишь запрещает изменять саму коллекцию через предоставленный API. 

Если нужно защитить элементы коллекции от изменений, то лучше сделать их неизменяемыми (immutable).

### Дополнительно

- если важно полностью защитить данные, возвращайте копию коллекции (ToArray(), ToList()), особенно в многопоточной среде;
  
- можно использовать AsReadOnly() для создания read-only-обёртки:

```public ReadOnlyCollection<Item> Items => _items.AsReadOnly();```

- избегайте возврата интерфейсов вроде IEnumerable<T>, если вызывающий код может привести коллекцию к изменяемому типу;

- для строгой неизменяемости используйте immutable-коллекции (```ImmutableList<T>```, ```ImmutableArray<T>``` из ```System.Collections.Immutable```);

Такие подходы помогают сохранить инкапсуляцию и избежать неожиданных изменений состояния объекта извне.

### Как отвечать на собеседовании

Если нужно вернуть коллекцию безопасно, не следует отдавать наружу изменяемый ```List<T>```, потому что внешний код сможет изменить внутреннее состояние объекта. Обычно используют ```IReadOnlyList<T>``` или ```ReadOnlyCollection<T>```, чтобы предоставить доступ только для чтения. Если требуется полностью изолировать состояние, возвращают копию коллекции или immutable-коллекцию.

</details>

<details>
<summary><b>Почему IEnumerable<T> не всегда лучший вариант для возврата коллекции?</b></summary>

>`IEnumerable<T>` описывает только возможность последовательного перебора элементов. Этот интерфейс не гарантирует неизменяемость, не означает, что возвращается снимок данных, и не сообщает, можно ли безопасно перечислять коллекцию несколько раз.

### В чем проблема `IEnumerable<T>`

- предоставляет только механизм обхода (`foreach`);
- не гарантирует, что данные не изменятся между перечислениями;
- может быть ленивым (`deferred execution`) — элементы вычисляются во время итерации;
- может скрывать дорогостоящие операции (например, повторные запросы к базе данных).
- вызывающий код может **перечислять несколько раз**, получая разные результаты;
- не даёт информации о размере коллекции (`Count`) без дополнительного прохода.

### Когда это важно

```csharp
public IEnumerable<Item> GetItems() => _items;
```

Из сигнатуры метода неясно:

- возвращается ссылка на внутреннюю коллекцию или независимая копия;
- безопасно ли перечислять результат несколько раз;
- изменятся ли данные между итерациями;
- насколько дорого повторное перечисление.
  
### Более явные альтернативы
- ```IReadOnlyList<T>``` — используется, когда коллекция уже существует в памяти и требуется предоставить доступ только для чтения.

``` csharp
public IReadOnlyList<Item> Items => _items;
```

- копия (снимок) — используется, когда важно вернуть снимок состояния на момент вызова.

``` csharp
public Item[] GetItems() => _items.ToArray();
```

IEnumarable<T> уместен, когда:

- действительно нужен ленивый обход;
- данные вычисляются по требованию;
- источник является потоковым;
- повторное перечисление не является обязательным.

### Важное уточнение

```IEnumerable<T>``` сам по себе не нарушает инкапсуляцию. Проблема возникает только тогда, когда через него возвращается ссылка на внутренний изменяемый объект.

``` csharp
// Потенциально опасно: возвращается тот же List<T>
public IEnumerable<Item> Items => _items;

// Безопасно: возвращается копия
public IEnumerable<Item> GetItems() => _items.ToArray();
```

### Итог

```IEnumerable<T>``` — это не «read-only коллекция», а контракт перечисления.

Если важно явно показать, что коллекцию нельзя изменять, обычно лучше использовать ```IReadOnlyList<T>``` или ```IReadOnlyCollection<T>```. Если требуется полностью изолировать внутреннее состояние, следует возвращать копию или immutable-коллекцию.

</details>

<details>
<summary><b>В чем разница между override и new?</b></summary>

> `override` переопределяет виртуальный метод (`virtual`, `abstract` или уже `override`) базового класса и участвует в полиморфизме.
>  
> `new` скрывает член базового класса. Какой член будет вызван, определяется по типу ссылки на этапе компиляции.

### Ключевое различие

- `override` — **динамическое связывание (runtime dispatch)**: вызывается реализация, соответствующая фактическому типу объекта;
- `new` — **статическое связывание (compile-time binding)**: вызывается член, определяемый типом переменной.

---

### Пример

```csharp
class Animal
{
    public virtual string Speak() => "Some sound";

    public string Eat() => "Animal eats";
}

class Dog : Animal
{
    public override string Speak() => "Woof"; // override

    public new string Eat() => "Dog eats"; // скрытие
}

Animal animal = new Dog();

Console.WriteLine(animal.Speak()); // Woof (полиморфизм)
Console.WriteLine(animal.Eat());   // Animal eats (тип ссылки Animal)

Dog dog = new Dog();
Console.WriteLine(dog.Eat());      // Dog eats
```

``` csharp
class Animal
{
    public virtual string Speak() => "Animal";
}

class Dog : Animal
{
    public override string Speak() => "Dog";
}

class Bulldog : Dog
{
    public new string Speak() => "Bulldog";
}

Animal animal = new Bulldog();
Dog dog = new Bulldog();
Bulldog bulldog = new Bulldog();

Console.WriteLine(animal.Speak());   // Dog
Console.WriteLine(dog.Speak());      // Dog
Console.WriteLine(bulldog.Speak());  // Bulldog

```
### Ключевая идея

- ```override``` продолжает виртуальную цепочку вызовов.
- ```new``` создаёт отдельный метод, который не участвует в полиморфизме

### Когда использовать
- **override** — когда нужно изменить поведение базового класса корректно и полиморфно;
- **new** — когда требуется намеренно скрыть реализацию базового класса (используется редко и осознанно).

### Важное замечание

```new``` не заменяет реализацию базового класса, а лишь скрывает её в производном классе. Это может приводить к неочевидному поведению, поэтому ```new``` используют только тогда, когда такое поведение действительно необходимо.

### Что говорить на собеседовании

Если после ```override``` в следующем производном классе используется ```new```, виртуальная цепочка обрывается. Для ссылок базовых типов продолжает вызываться последнее переопределение (```override```), а новый метод доступен только через ссылки конкретного типа.

</details>

<details>
<summary><b>Почему в C# нет множественного наследования классов?</b></summary>

> Множественное наследование классов приводит к конфликтам реализаций и состояния, а также к неоднозначности поведения (в частности, к *diamond problem*). Чтобы избежать этих сложностей, C# поддерживает наследование только от одного класса, но позволяет реализовывать несколько интерфейсов.

### В чем проблема множественного наследования

- **Конфликт реализаций** — если два базовых класса содержат методы с одинаковыми сигнатурами, возникает вопрос, какую реализацию следует использовать;
- **Конфликт состояния** — базовые классы могут содержать поля и свойства, что усложняет управление состоянием объекта;
- **Алмазная проблема (diamond problem)** — один и тот же предок может быть унаследован по разным путям, что приводит к неоднозначности поведения;
- **Сложность поддержки** — такие иерархии труднее понимать, тестировать и сопровождать;
- **Неочевидное поведение** — правила разрешения методов становятся менее предсказуемыми.

### Чем заменено в C#

**1. Реализация нескольких интерфейсов**

C# позволяет классу реализовывать несколько интерфейсов:

```csharp
interface IFlyable
{
    void Fly();
}

interface ISwimmable
{
    void Swim();
}

class Duck : IFlyable, ISwimmable
{
    public void Fly() => Console.WriteLine("Flying");
    public void Swim() => Console.WriteLine("Swimming");
}
```

**Преимущества:**

- интерфейсы (за исключением default implementations) не содержат состояния;
- класс сам определяет, как реализовать каждый контракт;
- отсутствует проблема наследования нескольких наборов полей.

**2. Композиция вместо наследования**

Предпочтительный подход — использовать композицию (```has-a```) вместо наследования (```is-a```), когда объект включает другие объекты и делегирует им часть поведения.

``` csharp
class Engine
{
    public void Start() => Console.WriteLine("Engine started");
}

class Car
{
    private readonly Engine _engine = new();

    public void Start() => _engine.Start();
}
```

Композиция уменьшает связанность и делает систему более гибкой.

**3. Реализации по умолчанию в интерфейсах (C# 8+)**

Интерфейсы могут содержать реализацию по умолчанию, однако:

- обычно не используются для хранения состояния;
- при конфликте реализаций требуют явного разрешения.

### Итог

C# не поддерживает множественное наследование классов, чтобы:

- избежать конфликтов реализаций и состояния;
- устранить неоднозначность поведения;
- упростить модель языка;
- сделать код более предсказуемым и удобным для сопровождения.

Вместо этого используются реализация нескольких интерфейсов и композиция как более безопасные и гибкие альтернативы.

</details>

<details>
<summary><b>Как связан LSP с наследованием?</b></summary>

> LSP гласит, что объекты производного класса должны быть взаимозаменяемы с объектами базового класса без нарушения корректности программы.
>
> Наследование считается корректным только в том случае, если объект производного класса можно использовать вместо объекта базового класса без изменения ожидаемого поведения программы.
> 
> Иными словами, **подкласс не должен нарушать контракт базового класса**.

---

### Классический пример нарушения LSP

```csharp
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}

public class Square : Rectangle
{
    private int _side;

    public override int Width
    {
        get => _side;
        set => _side = value;
    }

    public override int Height
    {
        get => _side;
        set => _side = value;
    }
}

Rectangle rectangle = new Square();
rectangle.Width = 2;
rectangle.Height = 3;

Console.WriteLine(rectangle.Area); // 9, хотя для Rectangle ожидается 6
```

### В чем проблема

Класс **Rectangle** предполагает, что ширина и высота изменяются независимо.

Класс **Square** нарушает это предположение: изменение одной стороны автоматически изменяет другую.

Следовательно, **Square** изменяет контракт базового класса **Rectangle**, и его нельзя безопасно использовать вместо **Rectangle**.

### Последствия нарушения LSP

- поведение становится непредсказуемым;
- код, работающий с базовым типом, начинает давать неверные результаты;
- наследование вводит скрытые ошибки;
- полиморфизм перестаёт быть безопасным.

### Вывод
- наследование оправдано только тогда, когда подкласс полностью сохраняет контракт базового класса;
- не всякое отношение «is-a» подходит для наследования;
- если подкласс меняет смысл операций или нарушает ожидания клиентского кода, это нарушение LSP;
- в таких случаях лучше использовать композицию или пересмотреть модель.

**LSP** — это основной критерий того, является ли выбранное наследование корректным с точки зрения проектирования.

</details>

<details>
<summary><b>Чем отличается раннее связывание от позднего?</b></summary>

> **Раннее связывание (early binding)** определяется на этапе компиляции.
>  
> **Позднее связывание (late binding)** определяется во время выполнения программы.

### Раннее связывание (compile-time)

Характеристики:

- решение о том, какой метод вызвать, принимается **на этапе компиляции**;
- зависит от **типа переменной**, а не объекта;
- работает быстрее, так как не требует дополнительной проверки во время выполнения.

Примеры:

- перегрузка методов (**overloading**);
- вызов невиртуальных методов;
- использование ключевого слова `new`.
- вызов статических методов.

---

### Позднее связывание (runtime)

Характеристики:

- окончательный выбор реализации происходит во время выполнения;
- зависит от фактического типа объекта или от информации, доступной только в runtime;
- используется для реализации полиморфизма.

Примеры:

- вызов `virtual` / `override` методов;
- работа через базовый тип:

```csharp
Animal animal = new Dog();
animal.Speak(); // определяется во время выполнения
``` 
использование dynamic:

``` csharp
dynamic obj = new Dog();
obj.Speak(); // метод определяется в runtime
``` 
### Важное уточнение

Следует различать два механизма позднего связывания:

- Виртуальная диспетчеризация (```virtual / override```) — выбор идёт по таблице виртуальных методов на основе фактического типа объекта.
- ```Dynamic binding (dynamic)``` — поиск подходящего члена выполняется DLR во время выполнения.

Оба механизма относятся к позднему связыванию, но реализованы по-разному.

### Ключевое различие
- раннее связывание → что вызвать известно заранее (compile-time);
- позднее связывание → что вызвать определяется в runtime.

### Итог

Раннее связывание обеспечивает простоту и предсказуемость, а позднее связывание позволяет реализовать полиморфизм и динамическое поведение.

</details>

<details>

<summary><b>Что такое vtable простыми словами?</b></summary>

> **vtable (virtual method table)** — это внутренняя структура данных, которую использует CLR для выбора нужной реализации виртуального метода во время выполнения.

### Как это работает

- для типов, содержащих виртуальные методы, CLR хранит информацию о соответствующих реализациях;
- в этой структуре находятся ссылки на актуальные версии виртуальных методов;
- при вызове метода через базовый тип CLR использует эту таблицу, чтобы определить, какую реализацию вызвать.

### Пример (идея)

```csharp
Animal animal = new Dog();
animal.Speak(); // вызывается Dog.Speak()
```

Хотя переменная имеет тип ```Animal```, будет вызван метод ```Dog.Speak()```, потому что:

- во время выполнения известен фактический тип объекта (```Dog```);
- CLR использует таблицу виртуальных методов для поиска нужной реализации.

### Зачем это нужно

- обеспечивает полиморфизм;
- позволяет вызывать разные реализации через общий базовый тип;
- делает выбор метода автоматическим и прозрачным для разработчика.

### Важно понимать

- ```vtable``` — это деталь реализации CLR, а не конструкция языка;
- разработчик напрямую с ней не работает;
- она используется только для виртуальных методов (```virtual```, ```override```, ```abstract```).

### Что говорить на собеседовании

```vtable``` — это внутренняя таблица, в которой CLR хранит ссылки на реализации виртуальных методов. Когда метод вызывается через базовый тип, CLR смотрит в эту таблицу и выбирает реализацию, соответствующую фактическому типу объекта.

</details>

<details>
<summary><b>Чем dynamic отличается от обычного полиморфизма?</b></summary>

> Обычный полиморфизм основан на общем контракте (базовом классе или интерфейсе) и позволяет вызывать разные реализации через один и тот же тип.
>
> `dynamic` откладывает разрешение членов и проверку типов до времени выполнения, поэтому обеспечивает большую гибкость, но лишает проверок компилятора.

### Ключевые отличия

- **Обычный полиморфизм**
  - опирается на общий контракт (`interface` или базовый класс);
  - компилятор проверяет существование методов и корректность вызовов;
  - ошибки обнаруживаются до запуска программы;
  - для виртуальной диспетчеризации используются `virtual` / `override` (или реализация интерфейсов);
  - выбор конкретной реализации происходит во время выполнения.

- **`dynamic`**
  - не требует общего контракта;
  - компилятор не проверяет наличие вызываемых членов;
  - разрешение методов и свойств происходит только в runtime;
  - ошибки возникают во время выполнения (`RuntimeBinderException`);
  - часто используется при работе с COM-объектами, динамическими API, JSON и reflection-подобными сценариями.

### Важное уточнение

Обычный полиморфизм сочетает:

1. **Статическую типизацию** — компилятор знает контракт и проверяет корректность вызовов;
2. **Позднее связывание** — конкретная реализация выбирается в runtime.

`dynamic` также использует позднее связывание, но без статической проверки контракта.

### Пример

```csharp
dynamic d = new Dog();
Console.WriteLine(d.Speak()); // вызов определяется в runtime
```

Если у объекта не окажется метода ```Speak```, программа завершится ошибкой во время выполнения.

### Пример обычного полиморфизма
``` csharp
Animal animal = new Dog();
Console.WriteLine(animal.Speak()); // Dog.Speak()
```

Здесь компилятор гарантирует, что метод ```Speak()``` существует в контракте ```Animal```, а CLR во время выполнения выбирает реализацию ```Dog```.

### Итог

dynamic и обычный полиморфизм оба используют позднее связывание, но различаются уровнем типовой безопасности:

- **обычный полиморфизм** — есть общий контракт и проверки на этапе компиляции;
- **dynamic** — контракт неизвестен компилятору, все проверки откладываются до runtime.

</details>

<details>
<summary><b>Бывает ли полиморфизм без наследования?</b></summary>

Да.

### Примеры

- через интерфейсы

Разные классы могут реализовывать один и тот же контракт, и код работает с ними через интерфейс:

``` csharp
interface IAnimal
{
    void Speak();
}
```
Это классический пример подтипного полиморфизма без наследования от конкретного класса.

- через обобщения (generics)

Один и тот же код может работать с объектами разных типов:

``` csharp
void Print<T>(T value)
{
    Console.WriteLine(value);
}
```

- через dynamic

``` csharp
dynamic animal = new Dog();
animal.Speak();
```

Метод определяется во время выполнения без статически известного контракта.

### Важное уточнение

С точки зрения теории типов выделяют разные виды полиморфизма:

- **подтипный полиморфизм (полиморфизм подтипов)** — наследование и интерфейсы;
- **параметрический полиморфизм** — обобщения (generics);
- **ad-hoc полиморфизм** — перегрузка методов;
- **динамический полиморфизм** — dynamic.

Поэтому ответ «полиморфизм бывает без наследования» абсолютно корректен.

### Вывод

**Полиморфизм** — более широкое понятие, чем наследование. Наследование является лишь одним из способов его реализации. В C# полиморфизм также достигается с помощью интерфейсов, обобщений, перегрузки методов и ```dynamic```.

</details>

<details>
<summary><b>Чем отличается override от overload?</b></summary>

- **`override` (переопределение)** — изменение реализации унаследованного *виртуального* метода.
- **`overload` (перегрузка)** — объявление нескольких методов с одинаковым именем, но разными сигнатурами.

---

### `override` (переопределение)

- используется в наследовании;
- требует, чтобы метод базового класса был объявлен как `virtual`, `abstract` или `override`;
- участвует в полиморфизме;
- конкретная реализация выбирается во время выполнения (runtime);
- сигнатура метода должна совпадать с сигнатурой базового метода.

```csharp
class Animal
{
    public virtual void Speak() => Console.WriteLine("Some sound");
}

class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof");
}
```

**```overload``` (перегрузка)**

- не требует наследования;
- методы отличаются параметрами (тип, количество, порядок);
- выбор подходящего метода происходит на этапе компиляции;
- возвращаемый тип не участвует в перегрузке.

``` csharp
class Printer
{
    public void Print(int x) { }
    public void Print(string s) { }
}
```

### Ключевое различие
```override``` → изменяет реализацию существующего виртуального метода в производном классе;
```overload``` → создаёт новые версии метода с тем же именем, но разной сигнатурой.

### Коротко
```override``` = один метод, разные реализации в иерархии наследования
```overload``` = одно и то же имя метода, разные сигнатуры.

</details>

<details>
<summary><b>New — это полиморфизм?</b></summary>

> Нет. Ключевое слово `new` скрывает член базового класса и **не участвует в полиморфизме в классическом смысле**.

### Почему это не полиморфизм

- при использовании `new` выбор метода происходит **на этапе компиляции**;
- вызываемый метод зависит от **типа переменной**, а не от реального типа объекта;
- отсутствует динамическое связывание (в отличие от `virtual` / `override`).

---

### Пример

```csharp
class Animal
{
    public void Speak() => Console.WriteLine("Animal");
}

class Dog : Animal
{
    public new void Speak() => Console.WriteLine("Dog");
}

Animal a = new Dog();
a.Speak(); // Animal

Dog d = new Dog();
d.Speak(); // Dog
```
### Что происходит

Хотя объект фактически имеет тип ```Dog```, переменная a объявлена как ```Animal```. Поскольку метод ```Speak()``` не является виртуальным, компилятор связывает вызов с методом ```Animal.Speak()```.

Метод ```Dog.Speak()``` будет вызван только тогда, когда переменная имеет тип ```Dog```.

### Вывод
- ```new``` → скрытие (hiding), статическое связывание;
- ```override``` → переопределение и полиморфизм;
- ```new``` не расширяет виртуальную цепочку методов.

Использовать ```new``` стоит только осознанно, так как он может приводить к неожиданному поведению.

</details>

<details>
<summary><b>Как ООП связано с SOLID?</b></summary>

> SOLID — это набор принципов объектно-ориентированного проектирования.
> 
> ООП предоставляет базовые механизмы (классы, инкапсуляцию, наследование, полиморфизм), а SOLID описывает, как применять эти механизмы так, чтобы код оставался гибким, расширяемым и удобным для сопровождения.

### Связь принципов

- **Инкапсуляция → SRP (Single Responsibility Principle)**  
  Объект должен отвечать только за одну зону ответственности и сам управлять своим состоянием.

- **Наследование → LSP (Liskov Substitution Principle)**  
  Подклассы должны корректно заменять базовые классы без изменения ожидаемого поведения.

- **Абстракция → DIP (Dependency Inversion Principle)**  
  Зависимости должны строиться на абстракциях (интерфейсах), а не на конкретных реализациях.

- **Полиморфизм → OCP (Open/Closed Principle)**  
  Система должна быть открыта для расширения, но закрыта для изменения — это достигается через полиморфизм и интерфейсы.

- **Интерфейсы → ISP (Interface Segregation Principle)**  
  Лучше иметь несколько узких интерфейсов, чем один универсальный.

### Важное уточнение

Это соответствие условно. Принципы **SOLID** не привязаны жёстко к отдельным механизмам ООП и часто используют сразу несколько концепций.

Например:

- **OCP** обычно опирается одновременно на абстракции, интерфейсы и полиморфизм;
- **DIP** тесно связан с абстракцией и инверсией зависимостей;
- **SRP** относится прежде всего к распределению ответственности, а не только к инкапсуляции.

### Итог

- ООП отвечает на вопрос: **«какие инструменты предоставляет язык?»**
- SOLID отвечает на вопрос: **«как применять эти инструменты при проектировании?»**

Иными словами, **SOLID** — это практические рекомендации по использованию возможностей ООП для построения более качественной архитектуры.

</details>

<details>
<summary><b>Почему public set не всегда плохо?</b></summary>

> Потому что всё зависит от модели. Для DTO, `record`-моделей, настроек или объектов без бизнес-инвариантов это может быть вполне допустимо. Проблема возникает там, где объект обязан защищать своё корректное состояние.

### Когда `public set` — это нормально

- **DTO (Data Transfer Object)** — объекты для передачи данных между слоями или сервисами;
- **конфигурации и настройки** — значения обычно заполняются автоматически при десериализации;
- **анемичные модели данных** — объекты, не содержащие бизнес-логики;
- **ORM-сущности** — в некоторых случаях открытые сеттеры используются для работы с Entity Framework Core и другими ORM;
- **`record`-типы** — часто используются как immutable/почти immutable модели;

В этих сценариях объект в основном служит для хранения и передачи данных, а не для защиты сложного состояния.

### Когда это становится проблемой

- есть **бизнес-инварианты** (например, баланс не может быть отрицательным);
- состояние должно изменяться только по определённым правилам;
- нарушение корректности данных может привести к ошибкам в логике.
- важно предотвратить неконсистентное состояние.

```csharp
class BankAccount
{
    public decimal Balance { get; set; } // плохо: можно установить отрицательное значение
}
```

### Как лучше
- использовать ```private set```, ```protected set``` или ```init```;
- изменять состояние через методы, отражающие бизнес-операции;
- выполнять валидацию перед изменением состояния.

``` csharp
class BankAccount
{
    public decimal Balance { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException();
        Balance += amount;
    }
}
```

### Итог

```public set``` — это инструмент, а не антипаттерн сам по себе. Он уместен для простых моделей данных, но нежелателен в доменных объектах, которые должны контролировать своё состояние и защищать инварианты.

</details>

<details>
<summary><b>Когда использовать лучше композицию вместо наследования?</b></summary>

Это не классический принцип ООП, но одно из ключевых правил проектирования:

**предпочитай композицию наследованию (composition over inheritance)**

Если поведение можно вынести в отдельный объект и подставлять через интерфейс — это чаще лучше, чем строить жёсткую иерархию классов.

### Когда композиция обычно предпочтительнее

- поведение должно изменяться во время выполнения;
- существует несколько независимых аспектов поведения, которые нужно комбинировать;
- между объектами скорее отношение **`has-a`**, чем **`is-a`**;
- наследование приводит к глубокой или запутанной иерархии;
- есть риск нарушения принципа подстановки Лисков (LSP);
- требуется слабая связанность и удобство тестирования.

### Когда наследование подходит лучше

- существует устойчивое и корректное отношение **`is-a`** (иерархия родственных объектов);
- производный класс действительно является специализированной версией базового;
- подкласс сохраняет контракт базового класса (LSP);
- общее поведение естественно выносится в базовый класс.

### Пример

```csharp
public interface IFlyBehavior
{
    void Fly();
}

public class FlyWithWings : IFlyBehavior
{
    public void Fly() { }
}

public class NoFly : IFlyBehavior
{
    public void Fly()
    {
        throw new NotSupportedException("This bird cannot fly.");
    }
}

public class Bird
{
    private readonly IFlyBehavior _flyBehavior;

    public Bird(IFlyBehavior flyBehavior)
    {
        _flyBehavior = flyBehavior;
    }

    public void Fly() => _flyBehavior.Fly();
}
```

### Что происходит
- ```Bird``` не содержит логику полёта напрямую;
- поведение делегируется объекту, реализующему ```IFlyBehavior```;
- реализацию можно легко заменить;
- новые варианты поведения добавляются без изменения класса ```Bird```.

### Замечание к примеру

Использование ```NoFly```, выбрасывающего ```NotSupportedException```, — лишь учебная иллюстрация. В реальной модели это может быть признаком неудачного контракта интерфейса, поскольку не все птицы умеют летать. Иногда лучше вообще не предоставлять метод ```Fly() ```для объектов, которые не поддерживают такое поведение.

### Почему композиция часто лучше
- уменьшает связанность (```loose coupling```);
- упрощает модульное тестирование;
- позволяет изменять поведение без изменения класса;
- предотвращает разрастание иерархий;
- хорошо сочетается с принципами SOLID, особенно OCP и DIP.

### Итог

Наследование подходит для стабильного и корректного отношения ```is-a```, позволяет выстроить иерархию родственных объектов для последующего применения полиморфизма.

Если требуется гибкость, возможность комбинировать или подменять поведение, композиция чаще является более удачным решением.

</details>
