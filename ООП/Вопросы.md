# Вопросы к собеседованию 

<details>
<summary><b>Чем инкапсуляция отличается от сокрытия данных?</b></summary>

<div style="margin-top: 16px;"></div>

> **Сокрытие данных** — это механизм ограничения доступа к внутреннему состоянию объекта (например, с помощью `private`, `protected`). Оно предотвращает прямое вмешательство извне.
>
> **Инкапсуляция** — более широкое понятие: объект объединяет данные и методы для работы с ними и самостоятельно контролирует корректность своего состояния.

### Ключевое различие

- сокрытие данных — это **инструмент (механизм)**;
- инкапсуляция — это **принцип проектирования**.

Сокрытие данных является частью инкапсуляции и используется для её реализации.

### Дополнение

Инкапсуляция не просто ограничивает доступ, но и:

- задаёт правила работы с данными (валидация, проверки);
- гарантирует, что объект всегда находится в корректном состоянии;
- скрывает детали реализации, предоставляя только необходимый интерфейс.

Пример:

```csharp
class BankAccount
{
    private double balance; // сокрытие данных

    public void Deposit(double amount)
    {
        if (amount > 0) // контроль состояния (инкапсуляция)
        {
            balance += amount;
        }
    }

    public double GetBalance()
    {
        return balance;
    }
}
```
</details>

<details>
<summary><b>Что обычно нарушает инкапсуляцию в C#?</b></summary>

- публичные поля;
- свойства с `public set`, если объект должен защищать инварианты;
- возврат изменяемых коллекций напрямую;
- выдача ссылки на внутреннее состояние через `ref return`.

### Дополнительно

- отсутствие валидации в сеттерах или методах, изменяющих состояние;
- использование `internal`/`protected` без необходимости, расширяющее область доступа;
- передача наружу ссылок на изменяемые внутренние объекты (например, массивы, списки);
- изменение состояния объекта извне через полученные зависимости (утечка инкапсуляции);
- чрезмерное использование `public` вместо более строгих модификаторов доступа;
- отсутствие неизменяемости там, где она уместна (`readonly`, `init`, immutable-объекты);
- раскрытие деталей реализации через интерфейс класса (например, лишние методы);
- использование статических полей для хранения состояния, доступного извне.

Такие практики делают объект уязвимым к неконтролируемым изменениям и усложняют поддержку кода.

</details>

## Как безопасно вернуть коллекцию?

```csharp
private readonly List<Item> _items = new();

// Нежелательно: снаружи можно менять коллекцию
public List<Item> Items => _items;

// Хорошо: только чтение
public IReadOnlyList<Item> Items => _items;

// Если нужна именно копия-снимок
public Item[] GetItems() => _items.ToArray();
```

### Важное уточнение

```IReadOnlyList<T>``` не делает объекты внутри коллекции неизменяемыми. Он только не даёт менять саму коллекцию через этот API.

### Дополнительно

- если важно полностью защитить данные, возвращайте копию коллекции (ToArray(), ToList()), особенно в многопоточной среде;
  
- можно использовать AsReadOnly() для создания read-only-обёртки:

```public ReadOnlyCollection<Item> Items => _items.AsReadOnly();```

- избегайте возврата интерфейсов вроде IEnumerable<T>, если вызывающий код может привести коллекцию к изменяемому типу;

- для строгой неизменяемости используйте immutable-коллекции (ImmutableList<T>, ImmutableArray<T> из System.Collections.Immutable);

- если элементы коллекции тоже должны быть защищены, делайте их неизменяемыми (immutable);

Такие подходы помогают сохранить инкапсуляцию и избежать неожиданных изменений состояния объекта извне.

## В чем разница между `override` и `new`?

> `override` переопределяет виртуальный метод базового класса и участвует в полиморфизме.
>  
> `new` скрывает член базового класса, и выбор метода зависит от типа ссылки, а не только от фактического типа объекта.

### Ключевое различие

- `override` — **динамическое связывание (runtime)**: вызывается метод реального типа объекта;
- `new` — **статическое связывание (compile-time)**: вызывается метод в зависимости от типа переменной.

---

### Пример

```csharp id="r4z8hp"
class Animal
{
    public virtual string Speak() => "Some sound";

    public string Eat() => "Animal eats";
}

class Dog : Animal
{
    public override string Speak() => "Woof"; // override

    public new string Eat() => "Dog eats"; // скрытие
}

Animal animal = new Dog();

Console.WriteLine(animal.Speak()); // Woof (полиморфизм)
Console.WriteLine(animal.Eat());   // Animal eats (тип ссылки Animal)

Dog dog = new Dog();
Console.WriteLine(dog.Eat());      // Dog eats
```

### Когда использовать
- **override** — когда нужно изменить поведение базового класса корректно и полиморфно;
- **new** — когда требуется намеренно скрыть реализацию базового класса (используется редко и осознанно).

### Важное замечание

Использование **new** часто приводит к неожиданному поведению и усложняет код, поэтому в большинстве случаев предпочтительнее **override**.

**Почему в C# нет множественного наследования классов?**

> Потому что оно усложняет модель языка: возникают конфликты реализации и неоднозначность поведения, в том числе известная *diamond problem*. В C# вместо этого используется множественная реализация интерфейсов.

### В чем проблема множественного наследования

- **Конфликт реализаций** — если два базовых класса содержат одинаковый метод, непонятно, какую реализацию должен использовать производный класс;
- **Алмазная проблема (diamond problem)** — один и тот же базовый класс может быть унаследован по разным путям, что приводит к дублированию или неоднозначности поведения;
- **Сложность поддержки** — код становится труднее понимать, тестировать и сопровождать;
- **Неочевидное поведение** — порядок разрешения методов может быть запутанным.

### Чем заменено в C#

**1. Интерфейсы**

C# позволяет реализовывать несколько интерфейсов:

```csharp
interface IFlyable
{
    void Fly();
}

interface ISwimmable
{
    void Swim();
}

class Duck : IFlyable, ISwimmable
{
    public void Fly() => Console.WriteLine("Flying");
    public void Swim() => Console.WriteLine("Swimming");
}
```

- отсутствует конфликт состояния;
- класс сам определяет реализацию поведения.

**2. Композиция вместо наследования**

Предпочтительный подход: has-a вместо is-a уменьшает связанность, делает систему гибче и проще в поддержке.

``` CSharp
class Engine
{
    public void Start() => Console.WriteLine("Engine started");
}

class Car
{
    private readonly Engine _engine = new();

    public void Start() => _engine.Start();
}
```

**3. Реализации по умолчанию в интерфейсах (C# 8+)**

Интерфейсы могут содержать реализацию по умолчанию, но:

- не содержат состояния;
- при конфликте требуют явного разрешения.

### Итог

C# отказался от множественного наследования классов, чтобы:

- избежать неоднозначности и конфликтов;
- упростить модель языка;
- сделать поведение программ более предсказуемым.

Вместо этого используются интерфейсы и композиция как более безопасные и гибкие альтернативы.

**Как связан LSP с наследованием?**

> Наследование считается корректным только в том случае, если объект производного класса можно использовать вместо объекта базового класса без изменения ожидаемого поведения программы.

Иными словами, **подкласс не должен нарушать контракт базового класса**.

---

### Классический пример нарушения LSP

```csharp
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}

public class Square : Rectangle
{
    private int _side;

    public override int Width
    {
        get => _side;
        set => _side = value;
    }

    public override int Height
    {
        get => _side;
        set => _side = value;
    }
}

Rectangle rectangle = new Square();
rectangle.Width = 2;
rectangle.Height = 3;

Console.WriteLine(rectangle.Area); // 9, хотя для Rectangle ожидается 6
```

### В чем проблема

Класс **Rectangle** предполагает, что ширина и высота могут изменяться независимо. Однако **Square** нарушает это ожидание: изменение одной стороны автоматически меняет другую.

### В результате:

- поведение становится непредсказуемым;
- нарушаются ожидания кода, работающего с базовым типом;
- подстановка (Square вместо Rectangle) приводит к ошибкам.

### Вывод

- не всякое отношение «является» (is-a) подходит для наследования;
- если подкласс меняет смысл или контракт базового класса — это нарушение LSP;
- в подобных случаях лучше использовать композицию или пересмотреть модель.

LSP помогает определить, где наследование уместно, а где — нет.

**Чем отличается раннее связывание от позднего?**

> **Раннее связывание (early binding)** определяется на этапе компиляции.
>  
> **Позднее связывание (late binding)** определяется во время выполнения программы.

### Раннее связывание (compile-time)

Характеристики:

- решение о том, какой метод вызвать, принимается **на этапе компиляции**;
- зависит от **типа переменной**, а не объекта;
- работает быстрее, так как не требует дополнительной проверки во время выполнения.

Примеры:

- перегрузка методов (**overloading**);
- вызов невиртуальных методов;
- использование ключевого слова `new`.

---

### Позднее связывание (runtime)

Характеристики:

- выбор метода происходит **во время выполнения**;
- зависит от **реального типа объекта**;
- лежит в основе полиморфизма.

Примеры:

- вызов `virtual` / `override` методов;
- работа через базовый тип:

```csharp
Animal animal = new Dog();
animal.Speak(); // определяется во время выполнения
``` 
использование dynamic:

``` csharp
dynamic obj = new Dog();
obj.Speak(); // метод определяется в runtime
``` 

### Ключевое различие
- раннее связывание → что вызвать известно заранее (compile-time);
- позднее связывание → что вызвать определяется в runtime.

### Итог

Раннее связывание даёт производительность и предсказуемость, позднее — гибкость и полиморфизм.

**Что такое vtable простыми словами?**

> **vtable (virtual table)** — это внутренняя таблица, с помощью которой среда выполнения (CLR) выбирает нужную реализацию виртуального метода во время выполнения.

### Как это работает

- для каждого класса с виртуальными методами создаётся таблица методов;
- в ней хранятся ссылки на актуальные реализации методов;
- при вызове метода через базовый тип используется эта таблица, чтобы определить, какой метод вызвать.

### Пример (идея)

```csharp
Animal animal = new Dog();
animal.Speak(); // вызывается Dog.Speak()
```

Хотя переменная имеет тип Animal, будет вызван метод Dog, потому что:

- во время выполнения определяется реальный тип объекта;
- CLR смотрит в vtable и находит нужную реализацию.

### Зачем это нужно

- обеспечивает полиморфизм;
- позволяет вызывать разные реализации через общий интерфейс;
- скрывает сложную логику выбора метода от разработчика.

### Важно понимать

- vtable — это деталь реализации CLR, а не конструкция языка;
- разработчик напрямую с ней не работает;
- она используется только для виртуальных методов (virtual, override, abstract).

**Чем `dynamic` отличается от обычного полиморфизма?**

> Обычный полиморфизм опирается на **контракт типа** (базовый класс или интерфейс) и виртуальные методы.
>  
> `dynamic` откладывает проверку типов и выбор вызываемого члена до **runtime**, поэтому даёт больше гибкости, но снижает безопасность.

### Ключевые отличия

- обычный полиморфизм:
  - проверяется на этапе компиляции;
  - требует общего контракта (наследование или интерфейс);
  - ошибки выявляются **до запуска**;
  - использует `virtual` / `override`.

- `dynamic`:
  - тип проверяется только **во время выполнения**;
  - не требует общего базового типа;
  - ошибки возникают **в runtime**;
  - удобен для работы с динамическими объектами (например, COM, JSON, reflection).

### Пример

```csharp
dynamic d = new Dog();
Console.WriteLine(d.Speak()); // вызов определяется в runtime
```

Если у объекта не окажется метода ```Speak```, программа завершится ошибкой во время выполнения.

## Бывает ли полиморфизм без наследования?

Да.

### Примеры

- через интерфейсы

Разные классы реализуют один контракт:

``` csharp
interface IAnimal
{
    void Speak();
}
```

- через обобщения (generics)

Код работает с параметрами типов:

``` csharp
void Print<T>(T value)
{
    Console.WriteLine(value);
}
```

- через dynamic
- 
Поведение определяется во время выполнения без строгого контракта.

### Вывод

**Полиморфизм** — это не только наследование. Это более общее понятие, включающее разные способы работы с разными типами через единый интерфейс или поведение.

## Чем отличается `override` от `overload`?

- **`override` (переопределение)** — изменение реализации унаследованного *виртуального* метода.
- **`overload` (перегрузка)** — создание нескольких методов с одинаковым именем, но разной сигнатурой.

---

### `override` (переопределение)

- используется в наследовании;
- требует `virtual`, `abstract` или `override` в базовом классе;
- работает через **полиморфизм (runtime)**;
- сигнатура метода **не меняется**.

```csharp
class Animal
{
    public virtual void Speak() => Console.WriteLine("Some sound");
}

class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof");
}
```

**```overload``` (перегрузка)**

- не требует наследования;
- методы отличаются параметрами (тип, количество, порядок);
- выбор метода происходит на этапе компиляции;
- возвращаемый тип не участвует в перегрузке.

``` csharp
class Printer
{
    public void Print(int x) { }
    public void Print(string s) { }
}
```

### Ключевое различие
```override``` → изменяет поведение унаследованного метода;
```overload``` → создаёт новые версии метода с тем же именем.

### Коротко
```override``` = один метод, разные реализации в иерархии
```overload``` = одно имя, разные параметры

## `new` — это полиморфизм?

> Нет. `new` скрывает член базового класса и **не участвует в полиморфизме**.

### Почему это не полиморфизм

- при использовании `new` выбор метода происходит **на этапе компиляции**;
- вызываемый метод зависит от **типа переменной**, а не от реального типа объекта;
- отсутствует динамическое связывание (в отличие от `virtual` / `override`).

---

### Пример

```csharp
class Animal
{
    public void Speak() => Console.WriteLine("Animal");
}

class Dog : Animal
{
    public new void Speak() => Console.WriteLine("Dog");
}

Animal a = new Dog();
a.Speak(); // Animal

Dog d = new Dog();
d.Speak(); // Dog
```

### Вывод
- ```new``` → скрытие (hiding), статическое связывание;
- ```override``` → полиморфизм, динамическое связывание.

Использовать ```new``` стоит только осознанно, так как он может приводить к неожиданному поведению.

### Как ООП связано с SOLID?

> **SOLID** — это развитие и уточнение идей ООП на уровне проектирования.  
> Если ООП даёт базовые механизмы (классы, наследование, полиморфизм), то SOLID объясняет, **как правильно их применять**.

### Связь принципов

- **Инкапсуляция → SRP (Single Responsibility Principle)**  
  Объект должен отвечать только за одну зону ответственности и сам управлять своим состоянием.

- **Наследование → LSP (Liskov Substitution Principle)**  
  Подклассы должны корректно заменять базовые классы без изменения ожидаемого поведения.

- **Абстракция → DIP (Dependency Inversion Principle)**  
  Зависимости должны строиться на абстракциях (интерфейсах), а не на конкретных реализациях.

- **Полиморфизм → OCP (Open/Closed Principle)**  
  Система должна быть открыта для расширения, но закрыта для изменения — это достигается через полиморфизм и интерфейсы.

- **Интерфейсы → ISP (Interface Segregation Principle)**  
  Лучше иметь несколько узких интерфейсов, чем один универсальный.

---

### Итог

- ООП отвечает на вопрос: **«какие есть инструменты?»**
- SOLID отвечает на вопрос: **«как ими правильно пользоваться?»**

Вместе они помогают писать гибкий, расширяемый и поддерживаемый код.

### Почему `public set` не всегда плохо?

> Потому что всё зависит от модели. Для DTO, `record`-моделей, настроек или объектов без бизнес-инвариантов это может быть вполне допустимо. Проблема возникает там, где объект обязан защищать своё корректное состояние.

### Когда `public set` — это нормально

- **DTO (Data Transfer Object)** — простые структуры для передачи данных;
- **`record`-типы** — часто используются как immutable/почти immutable модели;
- **конфигурации и настройки** — данные загружаются и затем используются;
- **анемичные модели** без бизнес-логики.

В этих случаях объект — это просто контейнер данных, и строгая инкапсуляция не требуется.

---

### Когда это проблема

- есть **бизнес-инварианты** (например, баланс не может быть отрицательным);
- состояние должно изменяться **контролируемо**;
- есть логика, зависящая от корректности данных;
- важно предотвратить неконсистентное состояние.

```csharp
class BankAccount
{
    public decimal Balance { get; set; } // плохо: можно установить отрицательное значение
}
```

### Как лучше
- использовать private set или init;
- изменять состояние через методы;
- добавлять валидацию.

``` csharp
class BankAccount
{
    public decimal Balance { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException();
        Balance += amount;
    }
}
```

### Итог

```public set``` — это не ошибка сам по себе. Он становится проблемой, когда нарушает инварианты и контроль над состоянием объекта.

### Почему `IEnumerable<T>` не всегда лучший вариант для возврата коллекции?

> Потому что он описывает только **способ обхода**, но не гарантирует ни неизменяемость, ни «снимок» данных. Если важно явно показать, что коллекцию нельзя менять, обычно лучше использовать `IReadOnlyList<T>`. Если нужен снимок на момент вызова — вернуть копию через `ToArray()`.

### В чем проблема `IEnumerable<T>`

- не гарантирует, что данные **не изменятся** во время обхода;
- может быть **ленивым (deferred execution)** — данные вычисляются при итерации;
- может скрывать дорогие операции (например, повторные запросы к БД);
- вызывающий код может **перечислять несколько раз**, получая разные результаты;
- не даёт информации о размере коллекции (`Count`) без дополнительного прохода.

---

### Когда это важно

```csharp
public IEnumerable<Item> GetItems() => _items;
```

вызывающий код не знает:

- это «живая» коллекция или снимок;
- можно ли безопасно итерироваться несколько раз;
- изменятся ли данные между вызовами.
  
### Более явные альтернативы
- ```IReadOnlyList<T>``` — если коллекция уже есть и её нельзя менять:

``` csharp
public IReadOnlyList<Item> Items => _items;
```

- копия (снимок) — если важно зафиксировать состояние:

``` csharp
public Item[] GetItems() => _items.ToArray();
```

- IEnumerable<T> — когда:
  - реально нужен ленивый обход;
  - источник данных потоковый или вычисляемый;
  - не важен повторный проход.

### Итог

```IEnumerable<T>``` — это не «read-only коллекция», а контракт перечисления.

Используйте его осознанно и выбирайте более конкретные типы, если важно явно выразить поведение данных.

## Когда использовать лучше композицию вместо наследования?

Это не классический принцип ООП, но одно из ключевых правил проектирования:

**предпочитай композицию наследованию (composition over inheritance)**

Если поведение можно вынести в отдельный объект и подставлять через интерфейс — это чаще лучше, чем строить жёсткую иерархию классов.

---

### Когда использовать композицию

- поведение должно **меняться во время выполнения**;
- есть несколько **независимых вариаций поведения** (комбинации);
- нет строгого отношения **`is-a`**, а скорее **`has-a`**;
- нужно избежать глубокой иерархии классов;
- есть риск нарушения **LSP** при наследовании;
- требуется **слабая связанность** и гибкость.

---

### Когда лучше наследование

- есть чёткое и устойчивое отношение **`is-a`**;
- поведение базового класса **логично и безопасно расширяется**;
- не требуется динамическая замена поведения;
- иерархия действительно отражает предметную область.

---

### Пример

```csharp
public interface IFlyBehavior
{
    void Fly();
}

public class FlyWithWings : IFlyBehavior
{
    public void Fly() { }
}

public class NoFly : IFlyBehavior
{
    public void Fly()
    {
        throw new NotSupportedException("This bird cannot fly.");
    }
}

public class Bird
{
    private readonly IFlyBehavior _flyBehavior;

    public Bird(IFlyBehavior flyBehavior)
    {
        _flyBehavior = flyBehavior;
    }

    public void Fly() => _flyBehavior.Fly();
}
```

### Здесь:

- Bird не наследуется от разных типов;
- поведение полёта можно менять и комбинировать;
- нет дублирования и жёсткой связи.

### Почему композиция часто лучше
- уменьшает связанность (```loose coupling```);
- упрощает тестирование;
- позволяет менять поведение без изменения класса;
- предотвращает «разрастание» иерархий;
- лучше соответствует принципам SOLID (особенно DIP и OCP).

> Наследование хорошо подходит для устойчивого отношения ```is-a```.
> Если нужна гибкость, замена поведения или комбинация возможностей — чаще лучше использовать композицию.
